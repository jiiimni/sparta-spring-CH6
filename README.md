# ☕ 커피숍 주문 시스템

> 내일배움캠프 Spring 3기 CH6 과제 - 다수 서버 환경에서도 안정적으로 동작하는 커피숍 주문 시스템

---

## 📌 ERD

```
users
- id        BIGINT PK
- point     INT

menus
- id        BIGINT PK
- name      VARCHAR
- price     INT

orders
- id           BIGINT PK
- user_id      BIGINT FK → users.id
- menu_id      BIGINT FK → menus.id
- amount       INT          (주문 시점 가격 스냅샷)
- created_at   DATETIME     (인기 메뉴 7일 집계 기준)
```

> orders.amount에 주문 시점 가격을 저장한 이유:
> 나중에 메뉴 가격이 변경되더라도 주문 당시 결제 금액이 유지되어야 하기 때문입니다.

## 📌 API 명세

| Method | URL | 설명 | 요청 Body | 응답 |
|--------|-----|------|-----------|------|
| GET | /api/menus | 메뉴 목록 조회 | - | 메뉴 목록 |
| PATCH | /api/users/{userId}/point | 포인트 충전 | amount | 충전 후 잔액 |
| POST | /api/orders | 주문/결제 | userId, menuId | 주문 정보 |
| GET | /api/menus/popular | 인기 메뉴 조회 (7일) | - | 인기 메뉴 3개 |

### 공통 응답 형식
```json
{
  "success": true,
  "data": {},
  "message": null

```

## 📌 설계 의도

### 1. 포인트 충전/차감 - 동시성 문제

**문제 상황**

다수의 서버·인스턴스 환경에서 동일 유저에게 동시에 포인트 충전/차감 요청이 들어오면,
각 서버가 같은 잔액을 읽고 덮어씌워 실제보다 적게 반영되는 race condition이 발생합니다.

**선택: 비관적 락 (Pessimistic Lock)**

| 전략 | 장점 | 단점 | 선택 여부 |
|------|------|------|-----------|
| 비관적 락 | 충돌 자체를 막아 정합성 확실히 보장 | 대기 시간 발생 | ✅ 선택 |
| 낙관적 락 | 충돌 없을 때 성능 좋음 | 충돌 시 재시도 로직 필요 | - |
| Redis 분산락 | 다중 서버 완벽 대응 | 인프라 추가 필요 | - |

**선택 이유**

포인트는 금융 데이터에 준하므로 성능보다 정합성이 최우선입니다.
`SELECT ... FOR UPDATE`로 트랜잭션이 끝날 때까지 다른 요청을 대기시켜
동시 요청이 와도 순차적으로 처리되도록 보장했습니다.

---

### 2. 주문/결제 - 데이터 일관성

**문제 상황**

포인트 차감과 주문 생성이 별개로 처리되면,
포인트는 차감됐는데 주문이 실패하는 상황이 발생할 수 있습니다.

**선택: 단일 트랜잭션 처리**

포인트 차감 → 주문 생성 → 데이터 플랫폼 전송을 하나의 트랜잭션으로 묶어,
중간에 실패하면 전체 롤백되도록 구현했습니다.

---

### 3. 인기 메뉴 조회 - 정확성

**문제 상황**

캐싱을 사용하면 성능은 좋지만 주문 횟수가 부정확할 수 있습니다.

**선택: 매 요청마다 DB 집계 쿼리 실행**

과제 요구사항에 "메뉴별 주문 횟수가 정확해야 한다"고 명시되어 있어,
정확성을 최우선으로 매 요청마다 7일치 주문을 COUNT로 집계합니다.
